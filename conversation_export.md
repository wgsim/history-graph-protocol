## 1. CLAUDE.md — 수동 메모리 (Manual Memory)

**알고리즘은 단순합니다: 세션 시작 시 파일을 읽어 컨텍스트 창에 직접 주입.**

CLAUDE.md 파일은 Claude Code가 매 세션 시작 시 자동으로 읽는 마크다운 파일로, 프로젝트 지침, 개인 워크플로우, 조직 전체 규칙을 저장합니다.

파일 로딩 전략은 계층적입니다. 현재 작업 디렉토리에서 루트까지 재귀적으로 탐색하며 `CLAUDE.md`와 `CLAUDE.local.md`를 찾습니다. 서브디렉토리의 메모리 파일은 Claude가 실제로 해당 디렉토리의 파일에 접근할 때만 로드되어, 불필요한 토큰 낭비를 방지합니다.

### 파일 위치별 우선순위

| 위치 | 용도 | 공유 여부 |
|---|---|---|
| `~/.claude/CLAUDE.md` | 글로벌 개인 설정 | 개인 전용 |
| `./CLAUDE.md` | 프로젝트 팀 규칙 | git 커밋됨 |
| `./CLAUDE.local.md` | 로컬 개인 메모 | gitignore |

---

## 2. Auto Memory — 자동 메모리 (Auto Memory)

**Claude가 스스로 학습하고 파일을 작성하는 시스템.**

각 프로젝트는 `~/.claude/projects/<project>/memory/` 에 전용 메모리 디렉토리를 가집니다. `<project>` 경로는 git 저장소에서 파생되므로, 같은 저장소 내 모든 worktree와 서브디렉토리가 하나의 auto memory 디렉토리를 공유합니다.

디렉토리 구조:
```
~/.claude/projects/<project>/memory/
├── MEMORY.md          # 매 세션 로드되는 인덱스
├── debugging.md       # 디버깅 패턴 상세 메모
├── api-conventions.md # API 설계 결정
└── ...                # Claude가 생성하는 토픽 파일들
```
`MEMORY.md`는 메모리 디렉토리의 인덱스 역할을 하며, Claude는 세션 내내 이 디렉토리의 파일을 읽고 씁니다.

### Auto Memory 생성 알고리즘

Claude Code는 반복되는 패턴이나 여러 번 적용되는 수정을 감지하면 `MEMORY.md`를 생성/업데이트합니다. 예를 들어:
```
# MEMORY.md (자동 생성)
- 이 프로젝트는 npm이 아닌 bun 사용
- E2E 테스트는 /tests/e2e/에 있으며 Playwright 사용
- 커밋 전 항상 `bun run typecheck` 실행
```
`MEMORY.md`는 시스템 프롬프트에서 200줄로 제한됩니다. 초과하는 내용은 로드되지 않습니다.

## 핵심 원리 요약

ChatGPT와 Claude는 근본적으로 다른 접근 방식을 취합니다. ChatGPT는 미리 요약을 계산해 모든 대화에 주입하는 반면, Claude는 메모리 도구를 즉석에서 사용합니다. CLAUDE.md 파일은 이 검색 문제를 완전히 우회합니다 — 선택적 검색이 아니라, 세션 시작 시 항상 컨텍스트 창에 직접 로드됩니다.

즉, Claude Code의 메모리는 **"AI 학습"이 아니라 "파일 시스템 + 컨텍스트 주입"** 이라는 단순하지만 효과적인 엔지니어링 패턴입니다.

## Auto Memory의 실제 스코프

메인 Claude Code 세션은 **프로젝트 로컬 메모리(MEMORY.md)만** 가집니다. 유저 전체 레벨의 auto memory는 일반 세션에 존재하지 않습니다.

정리하면:

| 메모리 타입 | 유저(글로벌) 레벨 | 프로젝트 레벨 |
|---|---|---|
| **수동 CLAUDE.md** | ✅ `~/.claude/CLAUDE.md` | ✅ `./CLAUDE.md` |
| **Auto Memory (MEMORY.md)** | ❌ 없음 | ✅ `~/.claude/projects/<project>/memory/` |

---

## 왜 유저 레벨 Auto Memory가 없을까?

설계 의도가 명확합니다:

- **Auto Memory의 목적** = 특정 프로젝트에서 Claude가 발견한 패턴(빌드 명령어, 파일 구조, 프레임워크 컨벤션)을 기억하는 것
- 이런 지식은 **프로젝트마다 다르기 때문에** 글로벌 레벨에 쌓는 게 의미가 없음
- 개인 전체 선호도(언어, 코드 스타일 등)는 **사용자가 직접 `~/.claude/CLAUDE.md`에 수동으로 작성**하는 것이 더 정확하다고 판단

---

## 예외: 커스텀 에이전트(서브에이전트)

Claude Code의 에이전트 위저드로 생성된 커스텀 에이전트에 한해 세 가지 스코프의 별도 시스템이 있습니다.
- **Project** — `.claude/agent-memory/<name>/` (git 커밋 가능, 팀 공유 가능)
- **Local** — `~/.claude/agent-memory-local/<name>/`
- **User** — `~/.claude/agent-memory/<name>/` (모든 프로젝트에 걸친 글로벌)

즉, **일반 Claude Code 세션**에서는 유저 레벨 auto memory가 없고, **커스텀 에이전트를 별도로 만들었을 때만** 유저 스코프 auto memory가 작동합니다.

## 글로벌 레벨 Auto Memory가 생긴다면 벌어질 일들

### 🟢 잠재적 이점

**크로스 프로젝트 패턴 학습**이 가능해집니다. 예를 들어 "이 유저는 항상 pnpm을 쓰고, TypeScript strict를 선호하며, 커밋 전에 lint를 돌린다"는 패턴을 A 프로젝트에서 배워서 B 프로젝트에 바로 적용할 수 있습니다. 지금은 이걸 `~/.claude/CLAUDE.md`에 수동으로 직접 써야만 가능합니다.

---

### 🔴 문제가 생기는 상황들

**1. 컨텍스트 오염 (Context Pollution)**

가장 심각한 문제입니다. 프로젝트별 지식이 다른 프로젝트에 침투합니다.

```
# 글로벌 MEMORY.md에 쌓인다면
- Project A: PostgreSQL 사용, port 5432
- Project B: MongoDB 사용, port 27017  
- Project A: 절대 CASCADE DELETE 쓰지 말 것
- Project C: Django ORM 사용
```

Project B에서 작업 중인데 Claude가 PostgreSQL 규칙을 들고 올 수 있습니다.

**2. 200줄 한계의 빠른 포화**

현재 MEMORY.md의 첫 200줄만 세션 시작 시 로드됩니다. 프로젝트가 10개만 넘어가도 글로벌 MEMORY.md는 금방 포화 상태가 되고, 오래된 프로젝트 정보가 밀려나거나 더 중요한 현재 프로젝트 정보가 잘립니다.

**3. 충돌 해소 불가능**

충돌 시 프로젝트 CLAUDE.md가 유저 파일보다 우선합니다. Auto memory는 재정의 없이 보완만 합니다. 글로벌 auto memory가 생기면 "프로젝트 A는 tabs, 프로젝트 B는 spaces"처럼 서로 상충하는 규칙이 같은 파일에 공존하게 됩니다. Claude가 어느 쪽을 따를지 예측이 불가능해집니다.

**4. 보안/프라이버시 리스크**

글로벌 메모리에 특정 프로젝트의 내부 정보(API 엔드포인트, 환경변수 패턴, 내부 아키텍처 결정)가 쌓이면, 전혀 다른 프로젝트 세션에서 그 정보가 노출될 수 있습니다. 특히 회사 업무용 프로젝트와 개인 프로젝트를 같은 머신에서 쓸 때 위험합니다.

---

### 왜 Anthropic이 이 설계를 선택했나

설계 원칙은 이렇습니다: 개인 상태나 크로스 프로젝트 조정은 글로벌에, 팀과 공유 가능한 프로젝트 설정은 프로젝트 레벨에 둡니다. Auto memory는 특정 프로젝트에 관한 것이지만 "개인적 학습"이기 때문에 글로벌에 저장됩니다 — 단 프로젝트별로 분리된 디렉토리에.

즉, "글로벌에 저장하되 프로젝트별로 격리"가 현재 구조의 핵심입니다.

---

### 현실적인 대안

크로스 프로젝트 지식, 즉 "API 프로젝트의 변경이 WordPress 플러그인에 영향을 준다"는 식의 관계 정보는 auto memory로 처리할 수 없습니다. 이를 위해서는 MCP Memory 서버(공유 지식 그래프)를 사용하는 것이 맞는 방식입니다. 동일한 MCP 서버에 연결된 모든 Claude 세션은 같은 엔티티와 관계를 볼 수 있습니다.

정리하면, 글로벌 레벨 auto memory는 **단기적으로 편해 보이지만 컨텍스트 오염, 토큰 포화, 충돌 예측 불가 문제**로 금방 무너지는 구조입니다. 그래서 Anthropic은 수동(`~/.claude/CLAUDE.md`) + 프로젝트별 격리 자동(MEMORY.md)이라는 현재 구조를 택한 것으로 보입니다.

## MCP Memory 서버란?

**Knowledge Graph Memory Server**는 로컬 지식 그래프를 통해 Claude가 채팅 간에 정보를 기억할 수 있게 해주는 MCP 서버입니다. Anthropic이 공식으로 만들었으며 오픈소스입니다.

---

## Auto Memory vs MCP Memory 비교

| | Auto Memory (MEMORY.md) | MCP Memory Server |
|---|---|---|
| 저장 형식 | 플랫 마크다운 텍스트 | **지식 그래프 (엔티티 + 관계)** |
| 스코프 | 프로젝트별 격리 | **크로스 프로젝트** |
| 작동 방식 | 세션 시작 시 자동 주입 | **Claude가 명시적 tool call로 조회** |
| 관계 표현 | 불가능 | **가능** ("A는 B에 의존한다") |
| 누가 씀 | Claude 자동 | Claude가 tool 호출로 명시적으로 |

---

## 핵심 데이터 구조

세 가지 요소로 구성됩니다.

**Entities** — 그래프의 주요 노드. 이름, 타입, observations를 가집니다.
**Relations** — 엔티티 간의 방향성 있는 연결. 항상 능동태로 저장됩니다.
**Observations** — 엔티티에 대한 개별 사실 정보로, 독립적으로 추가/삭제 가능합니다.

```json
{
  "entityName": "John_Smith",
  "observations": [
    "Speaks fluent Spanish",
    "Graduated in 2019",
    "Prefers morning meetings"
  ]
}
```


---

## 제공하는 Tool들

Claude가 호출할 수 있는 주요 도구들입니다.

- `read_graph` — 전체 지식 그래프 반환 (입력 불필요)
- `search_nodes` — 쿼리로 노드 검색. 엔티티 이름, 타입, observation 내용을 대상으로 검색
- `open_nodes` — 이름으로 특정 노드 조회, 관련 엔티티 간 관계도 함께 반환

---

## Auto Memory가 못 하는 것을 MCP Memory가 하는 이유

Auto memory는 관계를 모델링할 수 없습니다. 플랫 마크다운 노트를 저장할 뿐입니다. "이 API 엔드포인트는 저 인증 플로우에 의존한다"는 식의 표현은 타입이 있는 연결을 가진 엔티티가 필요합니다. 그것이 지식 그래프가 하는 일입니다. 또한 동일한 MCP 서버에 연결된 모든 Claude 세션은 같은 엔티티와 관계를 볼 수 있어 크로스 프로젝트 컨텍스트 공유가 가능합니다.

## 언제 쓰면 좋은가?

- **프로젝트 A와 B의 관계**를 기억시키고 싶을 때 ("A의 API가 바뀌면 B도 영향받는다")
- **사람, 팀, 의사결정** 같은 엔티티 간 관계를 장기 보존할 때
- **여러 머신**에서 같은 Claude 컨텍스트를 공유하고 싶을 때 (서버를 공유 위치에 두면 됨)
- Auto Memory의 200줄 한계를 넘는 방대한 프로젝트 지식을 관리할 때

네, 맞습니다. 전형적인 **방향성 있는 네트워크 그래프(Directed Graph)** 구조입니다.

---

## 구조 시각화

세 요소로 구성됩니다. **Entities**는 그래프의 주요 노드이고, **Relations**는 엔티티 간의 방향성 있는 연결이며 항상 능동태로 저장됩니다. **Observations**는 엔티티에 대한 개별 사실 정보입니다.

실제로 생긴 모양은 이렇습니다:

```
[auth-service] ──depends-on──▶ [database]
      │                              │
      │                         observations:
  connects-to                  - "PostgreSQL 16"
      │                        - "port 5432"
      ▼
[api-service] ──uses──▶ [JWT]
      │
      └──part-of──▶ [Project-A] ──uses──▶ [React 18.2]
                                      └──uses──▶ [Next.js 14]
```

하나의 노드가 **여러 노드와 동시에 관계**를 맺을 수 있고, 관계는 양방향이 아닌 단방향(directed)입니다.

---

## 실제 JSONL 저장 형태

파일에는 엔티티와 관계가 각 줄에 별도 JSON으로 저장됩니다:

```jsonl
{"type":"entity","name":"auth-service","entityType":"service","observations":["Uses JWT","Handles OAuth 2.0"]}
{"type":"entity","name":"database","entityType":"infrastructure","observations":["PostgreSQL 16","port 5432"]}
{"type":"relation","from":"auth-service","to":"database","relationType":"connects-to"}
{"type":"relation","from":"api-service","to":"auth-service","relationType":"depends-on"}
```

---

## Connected Graph인가?

반드시 connected graph일 필요는 없습니다. 프로젝트 별 의존성, 사람, 개념 등 서로 관련 없는 엔티티들도 독립적으로 존재할 수 있습니다. 즉 실제로는 **여러 개의 subgraph가 공존하는 형태**가 됩니다:

```
[사용자 선호]          [프로젝트 A 의존성]     [팀 구조]
     │                      │                    │
[pnpm 선호]──uses──▶[bun]  [React]◀──uses──[Next.js]  [팀원A]──works-on──▶[팀원B]
```

| | MEMORY.md | MCP Knowledge Graph |

## 현재 존재하는 해결 방식 3가지

### 1. AGENTS.md / CLAUDE.md — 가장 단순한 공유 레이어

모든 에이전트가 읽을 수 있는 파일을 프로젝트 루트에 두는 방식입니다. Claude Code는 `CLAUDE.md`를, Codex CLI는 `AGENTS.md`를 읽습니다. 일부 프레임워크는 두 플랫폼을 동시에 초기화하는 `--dual` 모드를 지원합니다.

```
프로젝트 루트/
├── CLAUDE.md     ← Claude Code가 읽음
├── AGENTS.md     ← Codex CLI가 읽음
└── .gemini/      ← Gemini CLI가 읽음
```

한계는 명확합니다. 사람이 수동으로 동기화해야 하고, "A가 오늘 결정한 것"이 자동으로 B에게 전달되지 않습니다.

---

### 2. MCP Memory 서버를 공유 허브로 쓰기

여러 에이전트가 동일한 프로젝트에서 공유 메모리, 태스크 큐잉, 의존성을 통해 조율되는 구조가 실험적으로 등장하고 있습니다.

개념은 이렇습니다:

```
Claude Code ─┐
Gemini CLI  ─┼──▶ [공유 MCP Memory 서버] ──▶ knowledge graph
Codex CLI   ─┘         (단일 JSONL 파일)
```

Claude Code가 "auth-service 리팩토링 완료, JWT 방식 변경"을 그래프에 쓰면, Gemini CLI가 다음 세션에서 그 노드를 읽어서 맥락을 이어받을 수 있습니다. **현재 MCP 프로토콜을 세 에이전트 모두 지원하기 때문에 이론적으로 가능합니다.**

---

### 3. 오케스트레이션 레이어 — 가장 진보된 접근

Claude Code가 Codex와 Gemini에게 MCO를 통해 태스크를 병렬로 디스패치하고, 결과를 종합하는 구조입니다. 호출하는 에이전트가 맥락을 정리하고 태스크를 할당하며, MCO가 여러 에이전트에게 동시에 팬아웃합니다.

## 공유 메모리에 뭘 저장하나

그래프 노드 타입이 "코드 지식"에서 **"작업 관리"** 로 확장됩니다.

**태스크 노드** — 누가 무엇을 하고 있는지
```json
{
  "entity": "task:auth-refactor",
  "type": "task",
  "observations": [
    "assigned_to: claude-code",
    "status: in_progress",
    "started: 2026-03-01T09:00",
    "files_locked: [auth.ts, jwt.ts]"
  ]
}
```

**결정 노드** — 왜 그렇게 했는지
```json
{
  "entity": "decision:keep-jwt",
  "type": "decision",
  "observations": [
    "made_by: claude-code",
    "reason: 세션 공유 구조상 교체 불가",
    "affects: auth-service, payment-service"
  ]
}
```

**충돌 방지 노드** — 파일 잠금
```json
{
  "entity": "file:auth.ts",
  "type": "file_lock",
  "observations": [
    "locked_by: claude-code",
    "locked_at: 2026-03-01T09:00",
    "purpose: auth-refactor"
  ]
}
```

## 실제 작동 흐름

구체적으로 검증된 패턴은 이렇습니다. 구현 → 검증 → 적대적 리뷰 → 커밋의 루프에서, 구현은 Codex/Gemini CLI에 위임하고 리뷰는 항상 작성자와 다른 모델이 수행합니다. 작성한 모델이 직접 리뷰하지 않는 교차 모델 리뷰 구조입니다.

```
1. Claude Code (오케스트레이터)
   → memory에 task:auth-refactor 생성
   → assigned_to: gemini-cli 기록
   → files_locked: [auth.ts] 기록

2. Gemini CLI
   → memory에서 task 조회
   → auth.ts 작업 시작
   → 완료 후 status: done, 결과 노드 기록

3. Codex CLI
   → memory에서 완료된 task 조회
   → "gemini가 작성한 코드"를 리뷰하는 역할
   → 리뷰 결과를 decision 노드로 기록

4. Claude Code
   → 모든 노드 조회 후 종합
   → 유저에게 보고
```

## 이 구조의 핵심 가치

**파일 충돌 방지** — A가 `auth.ts`를 작업 중이면 memory에 lock이 걸려있어서 B가 같은 파일을 건드리지 않습니다.

**결정 추적** — "왜 이 구조로 짰는가"가 decision 노드로 남아서 다음 에이전트가 같은 질문을 반복하지 않습니다.

**교차 검증** — 에이전트들이 결정과 인과 지식 그래프를 공유하고, 5ms 안에 컨텍스트를 검색할 수 있습니다. 같은 회사 모델끼리의 편향을 다른 회사 모델이 잡아줍니다.

## Beads가 구현한 것

Beads는 Git을 분산 데이터베이스로 활용하고, 작업을 명시적 의존성을 가진 DAG(Directed Acyclic Graph)로 모델링합니다. 에이전트가 "실행 가능한 작업"을 식별하고, 중앙 서버 없이 세션 간 컨텍스트를 유지합니다.

핵심 설계 원칙은 "Agent UX"입니다. API와 툴 출력이 LLM에게 토큰을 최소화하고 모호하지 않도록 최적화되어 있습니다. SQLite는 읽기 전용 캐시로만 쓰고, 진짜 소스 오브 트루스는 JSONL 파일입니다. 새 댓글을 추가하면 파일 끝에 줄을 추가하는 방식이라 Git 머지가 충돌 없이 처리됩니다.

---

## 지금 생각하시는 아이디어와의 차이

Beads가 구현한 것:
```
태스크 노드 ──blocks──▶ 태스크 노드
            ──parent──▶ 태스크 노드  
            ──discovered-from──▶ 태스크 노드
```

생각하시는 것은 한 단계 더 나아간 개념입니다:

```
[CLI 세션 시작]
    │
    ▼
[태스크 A] ──assigned_to──▶ [Claude Code]
    │           └── started_at, files_touched, decisions
    │
    ├──blocks──▶ [태스크 B] ──assigned_to──▶ [Gemini CLI]
    │
    └──discovered──▶ [태스크 C] ──assigned_to──▶ [Codex CLI]
                         └── reviewed_by: Claude Code (교차 검증)
```

**에이전트 정체성 + 결정 이력 + 파일 소유권**까지 그래프에 포함되는 구조입니다.

---

## 실제로 설계한다면

Beads의 의존성 그래프는 부모-자식 관계와 블로킹 의존성을 유지합니다. 에이전트는 태스크 A가 태스크 B를 블록한다는 것을 실제로 이해하고, A가 완료되기 전에 B를 시도하지 않습니다.

이걸 확장하면 노드 타입이 세 가지로 구분됩니다.

**Session 노드** — CLI를 시작한 순간
```jsonl
{"type":"session","id":"sess-a1b2","agent":"claude-code","started_at":"2026-03-01T09:00","trigger":"user"}
```

**Task 노드** — 실제 작업 단위
```jsonl
{"type":"task","id":"task-c3d4","title":"auth 리팩토링","status":"in_progress","assigned_to":"claude-code","session":"sess-a1b2","files_locked":["auth.ts"]}
```

**Decision 노드** — 에이전트가 내린 판단
```jsonl
{"type":"decision","id":"dec-e5f6","content":"JWT 유지","reason":"세션 공유 구조 변경 불가","made_by":"claude-code","affects_task":"task-c3d4"}
```

엣지는 이렇게 됩니다:
```jsonl
{"from":"task-c3d4","to":"task-g7h8","rel":"blocks"}
{"from":"task-c3d4","to":"dec-e5f6","rel":"produced"}
{"from":"sess-a1b2","to":"task-c3d4","rel":"initiated"}
{"from":"task-g7h8","assigned_to":"gemini-cli","rel":"handoff","from_agent":"claude-code"}
```

---

## Beads와 비교한 핵심 추가 요소

| | Beads | 생각하시는 구조 |
|---|---|---|
| 노드 | 태스크만 | 세션 + 태스크 + 결정 |
| 에이전트 구분 | assignee 필드 | 에이전트 정체성이 그래프의 1급 시민 |
| 이력 | 태스크 상태 변경 | **왜** 그 결정을 했는지까지 |
| 핸드오프 | 없음 | 에이전트 간 명시적 handoff 엣지 |
| 파일 충돌 방지 | 없음 | files_locked 노드 |

---

## 가장 중요한 설계 포인트

핵심은 이겁니다. "Beads가 에이전트에게 공유 메모리를 주고, Agent Mail이 메시징을 줍니다. 그게 전부입니다. 에이전트에게 태스크를 주고 알아서 해결하라고 하면 됩니다."

생각하시는 구조에서 추가로 해결해야 할 것은 **"누가 그래프에 쓸 권한이 있는가"** 입니다. 세 에이전트가 동시에 같은 노드를 업데이트하면 JSONL append-only 특성 덕에 충돌은 없지만, 어느 상태가 최신인지는 타임스탬프로 결정해야 합니다. Beads가 해시 기반 ID로 이 문제를 해결한 것처럼, 에이전트 정체성을 노드 ID에 포함시키는 방식 — 예를 들어 `claude-task-a1b2` vs `gemini-task-c3d4` — 이 실용적입니다.

                                          since: 09:05

## 그래서 뭐가 다른가

**에이전트 정체성이 1급 시민이다**

Beads에서 `assignee: claude-code`는 그냥 문자열 필드입니다. 조회하면 "누가 하고 있냐"는 알지만, claude-code가 이번 세션에서 어떤 다른 태스크들을 했는지, 어떤 파일을 건드렸는지는 별도로 연결이 없습니다.

생각하시는 구조에서는 에이전트 자체가 노드여서 그 에이전트가 생성한 결정, 건드린 파일, 핸드오프한 태스크가 전부 엣지로 연결됩니다. "claude-code 노드를 열면 이번 세션에서 한 모든 것이 딸려온다"는 게 가능해집니다.

**왜 그랬는지가 그래프에 남는다**

Beads는 태스크 상태 변화를 추적합니다. `open → in_progress → closed`

생각하시는 구조는 결정 노드가 따로 있어서 "JWT를 유지하기로 했다, 이유는 payment-service 의존성 때문"이 그래프에 남습니다. 나중에 Gemini CLI가 auth.ts를 수정하려 할 때, 연결된 결정 노드를 읽고 "이 파일은 이런 이유로 이 구조로 짜여있다"를 알 수 있습니다.

Beads에서는 이 정보가 태스크 description이나 comment에 텍스트로 묻혀있어서, 그래프 탐색으로 꺼낼 수 없고 LLM이 읽고 해석해야 합니다.

**핸드오프가 명시적 엣지다**

Beads에서 에이전트 전환은 `assignee` 필드 값을 바꾸는 것입니다. 누가 누구에게 왜 넘겼는지는 기록이 없습니다.

생각하시는 구조에서는 `handoff` 엣지 자체가 존재합니다. `from: claude-code, to: gemini-cli, reason: "컨텍스트 한계 도달"` 이 엣지를 따라가면 "이 태스크가 어떤 에이전트 경로를 거쳤는지"의 전체 궤적이 보입니다.

## 활용 방안

### 1. 디버깅 — "왜 이렇게 됐지?"

지금은 멀티 에이전트 작업이 잘못되면 원인을 찾기가 매우 어렵습니다. 에이전트가 명시적으로 로깅하도록 프로그래밍되지 않으면 추론 과정이 남지 않아서, 에이전트가 특정 결정을 내린 이유를 이해하거나 설명하기 어렵게 됩니다.

이력 그래프가 있으면 이게 가능해집니다:

```
"auth.ts가 왜 이 구조야?"
→ 그래프 탐색
→ Gemini가 09:23에 수정
→ 이유: Claude의 결정 노드 "JWT 유지" 참조
→ 그 결정의 이유: payment-service 의존성
```

근거를 추론하지 않고 **경로를 따라가는 것**으로 해결됩니다.

---

### 2. 세션 재개 — "어디까지 했지?"

Beads 창작자가 말하는 "50 First Dates 문제"입니다. 에이전트가 어제 한 작업을 기억하지 못하는 것이죠.

지금 방식은 에이전트가 컨텍스트를 잃으면 유저가 다시 설명합니다. 이력 그래프가 있으면 새 세션 시작 시 "지난 세션에서 어디까지, 누가, 왜 그렇게 했는지"를 그래프에서 읽고 바로 이어갑니다.

---

### 3. 교차 검증 — "이 결정이 맞아?"

Claude Code가 내린 결정을 Gemini CLI가 리뷰할 때, 그 결정의 맥락(왜 그랬는지)이 그래프에 있으면 더 정확한 리뷰가 가능합니다. 지금은 코드만 보고 판단하는데, 이력이 있으면 "이 결정이 당시 제약 조건 안에서 최선이었나"를 판단할 수 있습니다.

---

### 4. 거버넌스 — 규제 대응

EU AI Act는 고위험 AI 시스템에 대해 추적 가능성과 책임성 보장을 위한 자동 로깅을 명시적으로 요구합니다.

선도 조직들은 "bounded autonomy" 아키텍처를 구현하고 있습니다. 명확한 운영 한계, 고위험 결정에 대한 인간 에스컬레이션 경로, 그리고 에이전트 행동의 포괄적 감사 추적이 핵심입니다.

기업 환경에서 "AI가 왜 이 코드를 이렇게 짰나"를 감사관에게 설명해야 할 때 이 그래프가 그대로 증거가 됩니다.

## 유사 사례

### 소프트웨어 세계의 선례들

**Git 자체** — 코드 변경 이력을 DAG로 저장하는 것이 이미 검증된 패턴입니다. 생각하시는 것은 "코드 변경"이 아닌 "에이전트 행동과 결정"의 Git입니다.

**LangSmith (LangChain의 관측 도구)** — LLM 호출 이력을 추적하지만 에이전트 간 관계나 결정 맥락은 저장하지 않습니다.

**OpenTelemetry** — 분산 시스템에서 요청 추적을 트레이스 그래프로 저장합니다. 구조는 비슷하지만 "왜"가 없습니다.

### 가장 가까운 개념

OpenAI Agents SDK가 트레이싱을 에이전틱 앱 빌딩의 핵심 부분으로 명시하고 있습니다. 실용적인 규칙은 이렇습니다: 워크플로우가 안전 중심이면 그래프/상태 머신을 선호하고, 탐색적 리서치면 더 자유로운 멀티에이전트 패턴이 가능합니다.

## 결론적으로 어떤 의미인가

세 가지 레이어에서 의미가 있습니다.

**개발자 개인** — 멀티 에이전트 작업의 디버깅 시간을 줄이고, 세션이 끊겨도 컨텍스트를 잃지 않습니다.

**팀** — 누가 어떤 에이전트로 무슨 결정을 했는지가 공유 자산이 됩니다. 온보딩 비용이 줄고, 지식이 사람이 아닌 시스템에 남습니다.

**산업 전체** — 많은 조직들이 신뢰와 거버넌스를 파일럿을 넘어 에이전틱 AI를 확장하는 데 있어 핵심 장벽으로 꼽습니다. 실패를 검사할 수 없으면 안전하게 배포할 수 없습니다. 이 그래프가 그 장벽을 해결하는 구조적 답이 됩니다.

지금 이 아이디어가 빠른 이유는, Beads가 "할 일 관리"를 해결하고, LangSmith가 "LLM 호출 관측"을 해결했지만 **"멀티 에이전트 협업의 결정 이력"** 을 체계적으로 다루는 도구는 아직 없기 때문입니다.

## 왜 독립이 자연스러운가

오케스트레이터가 하는 일과 실무 에이전트가 하는 일은 근본적으로 다릅니다.

**실무 에이전트** — 코드를 읽고, 파일을 수정하고, 테스트를 실행합니다. 컨텍스트 윈도우를 코드와 작업 내용으로 가득 채웁니다. 깊이 파고드는 게 목적입니다.

**오케스트레이터** — "지금 전체 그림이 어떻게 되고 있나"를 봐야 합니다. 어떤 에이전트가 뭘 했고, 어떤 결정이 내려졌고, 다음은 누구에게 무엇을 맡겨야 하는지를 판단합니다. 이력 그래프를 읽고 쓰는 것 자체가 주된 작업입니다.

같은 모델이 두 역할을 동시에 하면, 깊은 작업을 하다가 컨텍스트가 오염되거나, 전체를 보려다 작업 집중도가 떨어집니다. 두 목적이 컨텍스트 윈도우를 두고 경쟁하는 상황이 됩니다.

---

## 독립시켰을 때 생기는 구조적 이점

이력 그래프와의 관계에서 특히 중요합니다.

오케스트레이터가 독립되어 있으면, 이력 그래프가 오케스트레이터의 **주 메모리**가 됩니다. 코드나 파일 내용이 아니라, "지금까지 어떤 결정이 어떤 순서로 내려졌나"가 컨텍스트의 전부입니다. 가볍고 명확합니다.

반면 실무 에이전트는 이력 그래프를 **참조만** 합니다. 작업 시작 전에 "내 태스크 노드와 연결된 결정 노드를 읽고" 작업하고, 완료 후 결과를 그래프에 씁니다. 그리고 자기 컨텍스트는 코드에 집중합니다.

---

## 그럼 어떤 모델이 오케스트레이터에 맞는가

실무 에이전트와 다른 특성이 필요합니다.

실무 에이전트는 코딩 능력, 긴 컨텍스트 처리, 도구 사용 정확도가 중요합니다. Claude Code나 Gemini CLI가 강점을 가진 영역입니다.

오케스트레이터는 코딩보다 **판단과 라우팅**이 핵심입니다. "이 결정의 영향이 어디까지 미치는가", "다음 작업은 어떤 에이전트가 적합한가", "충돌 위험이 있는 작업은 무엇인가"를 그래프를 읽고 판단합니다. 속도가 빠르고 추론이 명확한 모델이 오히려 맞습니다. 느리고 무거운 모델보다 빠른 판단에 최적화된 모델이 적합합니다.

---

## 완전 독립의 한 가지 리스크

오케스트레이터가 실무를 전혀 모르면 엉뚱한 라우팅을 할 수 있습니다. "이 태스크는 Gemini에게 맡겨라"고 결정했는데, Gemini가 현재 그 파일을 처리하기엔 컨텍스트가 부족한 상황이라는 걸 오케스트레이터가 알아야 합니다.

이 문제는 이력 그래프가 해결합니다. 오케스트레이터가 실무 내용을 직접 알 필요가 없고, 그래프에서 "Gemini의 현재 컨텍스트 상태", "어떤 파일이 어떤 에이전트에게 lock되어 있는지"를 읽으면 됩니다. 실무를 알지 않아도 상태를 아는 것만으로 충분합니다.

---

## 결론

독립이 맞습니다. 단, 완전한 단절이 아니라 **이력 그래프를 통해 간접적으로 연결된 독립**입니다.

```
오케스트레이터
    │ 읽기/쓰기
    ▼
[이력 그래프]
    ▲ 읽기/쓰기
    │
실무 에이전트들
```

직접 소통하지 않고 그래프가 중간 언어 역할을 합니다. 오케스트레이터는 그래프만 보고 판단하고, 실무 에이전트는 그래프에서 지시를 읽고 결과를 씁니다. 이렇게 되면 특정 에이전트가 바뀌거나 추가되어도 오케스트레이터 로직은 변경이 없습니다.

## 왜 병렬 CLI 구조에서 특히 필요한가

단일 CLI가 서브에이전트를 쓰는 경우는 그 CLI 내부에서 맥락이 유지됩니다. Claude Code가 서브에이전트를 띄워도 Claude Code가 전체를 알고 있습니다.

그런데 병렬 독립 CLI 구조는 다릅니다.

```
Claude Code ─── 서브에이전트A
                서브에이전트B

Gemini CLI  ─── 서브에이전트C

Codex CLI   ─── 서브에이전트D
                서브에이전트E
```

여기서 서브에이전트B와 서브에이전트C가 같은 파일을 건드리고 있는지, 서브에이전트A의 결정이 서브에이전트D에 영향을 주는지 — 이걸 아무도 모릅니다. 각 CLI는 자기 내부만 봅니다.

---

## 이력 그래프가 하는 역할

이 구조에서 그래프는 **CLI 경계를 넘는 유일한 공통 언어**가 됩니다.

```
Claude Code
  └─ 서브에이전트A ──┐
  └─ 서브에이전트B ──┼──▶ [이력 그래프] ◀──┬── 서브에이전트D ─┐
                     │                      └── 서브에이전트E ─┤ Codex CLI
Gemini CLI           │                                          │
  └─ 서브에이전트C ──┘                                          │
```

각 서브에이전트가 작업 시작 전에 그래프를 읽고, 완료 후 그래프에 씁니다. CLI 경계와 무관하게 전체 상태를 공유합니다.

---

## 서브에이전트가 동적으로 활성화될 때 특히 중요한 이유

서브에이전트는 필요에 따라 켜지고 꺼집니다. 이게 문제입니다.

서브에이전트C가 꺼졌다 다시 켜지면 이전에 뭘 했는지 모릅니다. 지금은 그 기억이 부모 CLI(Gemini)의 컨텍스트에만 있는데, 컨텍스트가 길어지면 희석됩니다.

이력 그래프가 있으면 서브에이전트C가 재활성화될 때 자기 노드를 읽고 "내가 어디까지 했고, 어떤 결정을 했고, 뭐가 아직 남아있는지"를 즉시 복원합니다.

---

## 실제로 문제가 될 수 있는 시나리오

**충돌 시나리오** — Claude Code 서브에이전트B가 `auth.ts` 수정 중인데, Gemini CLI가 새 태스크를 받고 서브에이전트C를 활성화해서 같은 파일을 건드리려 합니다. 그래프에 lock 노드가 없으면 두 에이전트가 동시에 같은 파일을 수정합니다.

**중복 작업 시나리오** — Codex 서브에이전트D가 JWT 구현을 완료했는데, Claude Code가 이 사실을 모르고 서브에이전트A에게 같은 작업을 할당합니다.

**결정 역행 시나리오** — 서브에이전트A가 "PostgreSQL 유지"를 결정했는데, 나중에 활성화된 서브에이전트E가 그 결정을 모르고 MongoDB로 마이그레이션을 시작합니다.

세 가지 모두 그래프에 상태가 기록되어 있으면 시작 전에 막을 수 있습니다.

---

## 오케스트레이터의 위치

이 구조에서 오케스트레이터는 CLI들 위에 있는 게 맞습니다.

```
        오케스트레이터
        (이력 그래프 관리 + 라우팅)
              │
    ┌─────────┼─────────┐
    ▼         ▼         ▼
Claude     Gemini     Codex
Code       CLI        CLI
  │          │          │
 서브       서브       서브
에이전트   에이전트   에이전트
(내부 자율) (내부 자율) (내부 자율)
```

오케스트레이터는 각 CLI 내부의 서브에이전트 동작에 간섭하지 않습니다. 각 CLI가 내부적으로 서브에이전트를 어떻게 쓰는지는 자율입니다. 오케스트레이터는 CLI 레벨 태스크 할당과 그래프 상태 관리만 합니다.

서브에이전트가 그래프에 쓸 때는 부모 CLI를 통해서가 아니라 직접 MCP Memory 서버에 씁니다. 이렇게 해야 CLI가 죽어도 서브에이전트의 이력이 남습니다.

---

## 결론

쓸만한 수준이 아니라, 병렬 독립 CLI + 동적 서브에이전트 구조에서 이 시스템이 없으면 규모가 커질수록 통제가 불가능해집니다. CLI 내부는 각자 알아서 하되, CLI 경계를 넘는 조율은 이력 그래프만이 할 수 있습니다.

## 단일 CLI 내부에서 충돌 방지 메커니즘

Claude Code 기준으로 실제 구현된 방식은 세 가지 레이어입니다.

---

### 레이어 1 — 파일 기반 잠금 (File Lock)

여러 에이전트가 동시에 같은 코드베이스에서 작업할 때 충돌 없이 진행하기 위해 `/coordination/` 디렉토리 구조를 생성합니다.

```
/coordination/
├── active_work_registry.json  # 현재 작업 중인 것 전체 목록
├── completed_work_log.json    # 완료된 작업 로그
├── agent_locks/               # 에이전트별 잠금 파일
│   └── {agent_id}_{timestamp}.lock
└── planned_work_queue.json    # 예정된 작업 대기열
```


에이전트가 파일을 건드리기 전에 lock 파일을 만들고, 다른 에이전트는 lock이 있으면 대기하거나 다른 작업으로 이동합니다.

---

### 레이어 2 — DAG 기반 의존성 실행 순서

태스크는 플랫 리스트가 아닌 DAG를 지원합니다. 태스크가 명시적으로 다른 태스크를 "블록"할 수 있습니다. 태스크 3(테스트 실행)은 태스크 1(API 빌드)과 태스크 2(인증 설정)가 완료되기 전에 시작할 수 없다는 것을 시스템이 강제합니다. 이것이 LLM 워크플로우에서 흔한 "완료 환각" 오류를 방지합니다.

순서가 강제되기 때문에 같은 파일에 동시 접근 자체가 구조적으로 방지됩니다.

---

### 레이어 3 — 에이전트 간 직접 메시징

태스크 파일과 SendMessage가 유일한 조율 채널입니다. 공유 메모리가 없습니다. 팀원들은 직접 서로 메시지를 보내고, 공유 태스크 목록에서 태스크를 자기 주장(claim)하며 협력합니다. 파일 잠금이 이중 claim을 방지합니다.

---

## 그런데 이 방식의 한계

각 팀원은 자신만의 컨텍스트 윈도우를 가진 완전한 Claude Code 세션입니다. 팀 리더의 대화 이력을 상속하지 않습니다.

여기서 핵심 문제가 드러납니다.

```
지금 Claude Code 내부 조율:
에이전트A ──메시지──▶ 에이전트B  (실시간 조율 O)
에이전트A ──────────▶ [?]        (왜 그 결정을 했는지 기록 X)
세션 종료 후 ─────────▶ [?]       (이력 없음 X)
```

즉, **지금 이 순간의 충돌은 막지만 이력은 남지 않습니다.**

---

## 이 맥락에서 이력 그래프의 포지션

단일 CLI 내부 조율이 해결하는 것과 이력 그래프가 해결하는 것이 다른 레이어입니다.

| | 단일 CLI 내부 조율 | 이력 그래프 |
|---|---|---|
| 목적 | 지금 이 순간 충돌 방지 | 지나온 결정과 맥락 보존 |
| 시간축 | 현재 | 과거 → 현재 → 미래 세션 |
| 범위 | 단일 CLI 내부 | CLI 경계 초월 |
| 소멸 | 세션 종료 시 사라짐 | 영구 보존 |

그래서 두 시스템이 충돌하는 게 아니라 **상호 보완**입니다.

```
단일 CLI 내부:
에이전트A + 에이전트B ──lock/DAG/메시징──▶ 현재 충돌 방지

이력 그래프:
그 과정에서 일어난 결정들 ──기록──▶ 다음 세션, 다른 CLI가 참조
```

단일 CLI는 실시간 교통 정리를 하고, 이력 그래프는 그 교통 기록을 블랙박스로 남깁니다. 두 레이어가 함께 있어야 완전한 시스템이 됩니다.

## 🤖 Claude

좋은 질문입니다. 몇 가지 각도에서 생각해볼 수 있어요.

**기술적 이유 — 설계 철학의 차이**

Claude Code의 현재 구조는 태스크 파일과 SendMessage가 유일한 조율 채널이고, 공유 메모리가 없습니다. 이건 의도적 선택입니다. Anthropic은 "UNIX 철학" — 단순하고, 예측 가능하고, 각 도구가 한 가지만 잘 하는 — 을 지향합니다. 이력 그래프는 복잡한 상태를 관리하는 시스템이고, 이 복잡성이 버그와 예측 불가능한 동작을 만들 수 있습니다.

**소유권 문제** — 이력 그래프를 누가 소유하는가가 불명확합니다. Claude Code 내부에 넣으면 Gemini CLI나 Codex CLI와는 공유가 안 됩니다. Anthropic 입장에서는 자사 제품 범위를 벗어난 문제입니다.

**제품 전략 — MCP를 통해 위임하는 전략** — Anthropic은 MCP라는 표준 프로토콜을 만들고, 이런 확장 기능은 생태계가 알아서 만들도록 설계했습니다.

**근본적 이유** — 아직 멀티 에이전트 자체가 실험 단계입니다. 이력 그래프는 멀티 에이전트가 안정화된 이후에 필요한 레이어입니다.

완전히 동일한 형태로 내부 테스트 중일 가능성은 낮지만, 구성 요소들은 이미 내부에 존재합니다. 가장 가능성 높은 시나리오는 Agent Teams가 stable로 전환된 이후 이력 그래프 레이어가 다음 단계로 구현될 가능성입니다.

## 🤖 Claude

정확히 짚으셨습니다. 핵심은 "필수 로딩" vs "필요시 탐색"의 차이입니다.

현재 메모리 시스템들의 문제: CLAUDE.md는 무조건 전체 로딩(비효율), MCP Memory는 수동 쿼리 필요(자동화 안됨). 이력 그래프는 작업 시작 전 자동 탐색 후 선택적 로딩이 가능합니다.

**표준화 필요성 (HGP — History Graph Protocol)**

MCP가 "도구 연결"을 표준화했다면, 다음 표준화 대상은 "에이전트 이력과 방향성 보존"입니다. 표준화되어야 할 것들: 노드/엣지 스키마(공통 포맷), 탐색 인터페이스(모든 에이전트가 동일하게 호출), 라이프사이클 훅(on_task_start, on_decision, on_task_end, on_handoff).

**순차적 도구 사용에서의 가치**

Day 1 Claude Code로 DB 스키마 설계 → Day 3 n8n으로 API 자동화 시 HGP 탐색으로 DB 결정 발견 → Day 7 Gemini CLI로 테스트 작성 시 soft delete 패턴 자동 반영. 이 탐색 없이는 각 도구가 매번 "처음 보는 프로젝트"처럼 접근합니다.

MCP가 인프라 레이어라면, HGP는 그 위의 시맨틱 레이어입니다. 충돌이 아니라 적층 구조입니다.
